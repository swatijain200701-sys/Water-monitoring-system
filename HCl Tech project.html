Core Algorithm & Logic

The system utilizes a time-series evaluation algorithm to differentiate between normal usage and abnormal wastage (leakage).

Algorithm Steps

  1. Initialize system parameters (flow_rate = 0, total_usage = 0, timer = 0).
  2. Read sensor data: Fetch input from the water flow sensor continuously.
  3. Evaluate state: If flow_rate > 0, increment total_usage.
  4. Threshold Check (Usage): If total_usage limits are exceeded, flag as High Usage.
  5. Continuous Flow Check (Leakage):
    • Start a timer when flow begins.
    • If flow stops, reset timer.
    • If flow continues non-stop for duration > LEAK_THRESHOLD_TIME, trigger Leakage Alert.
  6. Actuation: If alert is critical, send "Close Valve" command to solenoid valve.

Pseudocode


// Configuration
const THRESHOLD_TIME = 15 // seconds
const USAGE_LIMIT = 500   // Liters

// State Variables
float flow_rate = 0;
float total_vol = 0;
int continuous_flow_time = 0;
boolean valve_open = true;

function monitor_water():
  while (valve_open):
    flow_rate = read_sensor("YF-S201")
    
    if (flow_rate > 0):
      continuous_flow_time += 1
      total_vol += (flow_rate / 60)
      
      // Leakage Detection Logic
      if (continuous_flow_time > THRESHOLD_TIME):
        trigger_alert("Possible Leakage Detected!")
        valve_open = false // Shut off valve
        calculate_water_saved()
        break
        
      // Overuse Logic
      if (total_vol > USAGE_LIMIT):
        trigger_alert("High Usage Warning")
        
    else:
      continuous_flow_time = 0 // Reset on stop
      
    delay(1000) // 1 sec interval

System Architecture

Designed for easy transition from this software prototype to a physical IoT deployment.

1. Hardware Layer

Sensors: YF-S201 flow sensor for measuring water velocity. Ultrasonic sensor (HC-SR04) for tank level.

Actuator: 12V Solenoid Water Valve to cut off supply upon leakage detection.

Microcontroller: ESP32 or ESP8266 Wi-Fi module for local processing and network communication.

2. Communication Layer

Protocol: MQTT (Message Queuing Telemetry Transport) for lightweight, fast data publishing, or simple REST APIs via HTTP POST requests.

Payload: JSON objects containing timestamp, device ID, flow rate, and valve status.

3. Application Layer

This Dashboard: Built with HTML/CSS/JS. Parses incoming data, applies the time-series algorithm, renders Chart.js visualizations, and calculates analytics.

No complex backend required—can run fully in-browser utilizing LocalStorage or connected to a cloud broker.

Design Thinking Process

1. Empathize

Problem context: Hostels, colleges, and homes experience massive water loss due to unmonitored leaks or over-usage (e.g., leaving a tap running, broken pipes).

User Needs: Users need a way to know immediately if water is being wasted without having to constantly check physically. Administrators need to track overall billing/usage.

2. Define

Problem Statement: "How might we create an automated, real-time tracking system to detect continuous abnormal water flow and proactively prevent wastage in localized plumbing channels?"

3. Ideate

Ideas generated: Manual check-ins, mechanical auto-shutoff valves, smart shower heads.
Selected Idea: An inexpensive IoT add-on (Flow sensor + Solenoid Valve + Web Dashboard) that fits onto existing pipes, using a simple time-threshold algorithm to define "wastage".

4. Prototype

Current Iteration: A highly interactive web-based simulator (this dashboard). It models the expected behavior of the physical sensors and demonstrates the logic used to determine leakage vs. normal usage.

5. Test

Using the prototype "Simulation modes" to validate the logic. Can the system correctly trigger an alert when flow is continuous for too long? (Yes, use the "Simulate Leakage" button to test).

Project Evaluation Details

Innovation Highlights

  • Algorithmic simplicity: Uses a time-series threshold rather than complex machine learning, meaning it can run on underpowered microcontrollers.
  • Actionable insights: Doesn't just show data; it mathematically calculates "Water Saved" based on the time the valve is closed after an alert.
  • Device Agnostic: Web-based dashboard accessible from mobile, tablet, or desktop.

Future Scope

  • Integration with machine learning APIs to learn household specific "normal" water usage patterns.
  • Mobile app with Push Notifications via Firebase.
  • Multi-node support (tracking kitchen tap, bathroom, and main line on the same dashboard).
  • Gamification: Rewarding users with points for water conservation.
